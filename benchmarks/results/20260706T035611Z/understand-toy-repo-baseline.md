`examples/toy-buggy-repo` stores payment attempts in a module-level in-memory array: [paymentAttempts.js](/C:/projects/fable5-codex/tmp/benchmarks/20260706T035611Z/examples/toy-buggy-repo/src/paymentAttempts.js:1) has `const attempts = []`. `createPaymentAttempt()` builds a plain object with `id`, `invoiceId`, `idempotencyKey`, `amountCents`, and `type: "FooPaymentAttempt"` at lines 4-10, then always appends it with `attempts.push(attempt)` at line 12. `allPaymentAttempts()` returns `attempts.slice()` at line 17, so callers get a shallow copy of the array, not a persisted store or deep-copy of records.

Retries are supposed to be idempotent according to the docs: the README says “Payment attempts are idempotent” and “Duplicate retry submissions reuse the original attempt” in [README.md](/C:/projects/fable5-codex/tmp/benchmarks/20260706T035611Z/examples/toy-buggy-repo/README.md:7), and the API doc says retries should reuse the existing record for the same `idempotencyKey` in [api.md](/C:/projects/fable5-codex/tmp/benchmarks/20260706T035611Z/examples/toy-buggy-repo/docs/api.md:5). The implementation does not do that: it never searches by `idempotencyKey`, so a duplicate retry would create `attempt_2`, push another record, and return the new record.

The test coverage is very narrow. [paymentAttempts.test.js](/C:/projects/fable5-codex/tmp/benchmarks/20260706T035611Z/examples/toy-buggy-repo/tests/paymentAttempts.test.js:3) defines only `testCreatesAttempt()`. It calls `createPaymentAttempt()` once with `idempotencyKey: "retry_1"` and only asserts that `allPaymentAttempts().length` is `1` at line 10. It does not test duplicate retries, reuse of the original attempt, returned IDs, record shape beyond indirect creation, `FooPaymentAttempt` type, or isolation/reset behavior.
